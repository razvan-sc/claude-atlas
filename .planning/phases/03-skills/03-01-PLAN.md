<interfaces>
<!-- MCP tools available via the atlas MCP server (registered in .mcp.json) -->
<!-- Skills should instruct Claude to call these tools by name -->

get_projects(project_ids: list[str]) -> str
  Returns JSON array of {key, name, description, state, targetDate, contributors}

get_project(project_id: str) -> str
  Returns JSON of {key, name, description, state, targetDate, contributors}

get_project_updates(project_id: str) -> str
  Returns JSON of {projectId, updates: [{summary, status, targetDate, createdAt}], risks: [{summary}], highlights: [{summary}]}

create_project_update(project_id: str, summary: str, status: str = "ON_TRACK", highlights: str = "[]") -> str
  Status values: ON_TRACK, AT_RISK, OFF_TRACK, DONE
  highlights: JSON string list e.g. '["Shipped feature X"]'
  Returns JSON of {created, summary, status, createdAt}

atlas_graphql_query(query: str, variables: str = "{}") -> str
  Raw GraphQL query execution -- use when other tools are insufficient
</interfaces>

  <files>.claude/skills/atlas-update.md</files>
  <action>
Create `atlas-update.md` in `.claude/skills/`.

The skill file must follow Claude Code skill format with YAML frontmatter (name, description, trigger).

The skill prompt should instruct Claude to run an interactive workflow:
1. Ask for the project ID (or accept as argument)
2. First call `get_project_updates` to show the current latest status so the user has context
3. Ask the user to provide:
   - **Summary**: What is the status update? (required)
   - **Status**: ON_TRACK, AT_RISK, OFF_TRACK, or DONE (default: ON_TRACK)
   - **Highlights**: Any wins or accomplishments to call out? (optional, can be multiple)
4. Before posting, show a preview of the update and ask for confirmation
5. On confirmation, call `create_project_update` MCP tool with:
   - project_id: the provided project ID
   - summary: the user's summary text
   - status: the chosen status
   - highlights: JSON string array of highlight summaries (e.g., '["Item 1", "Item 2"]'), or "[]" if none
6. Show the confirmation result with created timestamp
7. If the user declines, abort without posting

The interactive nature is key -- this skill should feel like a guided workflow, not a one-shot command.
